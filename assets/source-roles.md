# Source Roles

This pack uses four local sources with distinct roles. The purpose is not to summarize them evenly; it is to extract a reusable motivation architecture.

## 1. 《遊戲化實戰全書》

**Role**: theory spine and terminology canon.

Use it for:

- accurate definitions of CD1-CD8
- left/right, white/black, explicit/implicit distinctions
- first-level Octalysis analysis
- four journey stages
- from-zero design process
- warnings about PBL, overjustification, black-hat fatigue, and ethics

Do not use it for:

- author persona imitation
- a complete game technique encyclopedia
- treating every chapter as equal summary material
- copying long examples into the pack

## 2. Section I: Octagon Framework And Motivation Basics

**Role**: diagnostic language.

Use it for:

- human-focused vs function-focused design
- core-drive diagnosis questions
- missing/overused drive signals
- left/right intrinsic-extrinsic tradeoff
- white/black long-term vs short-term tradeoff
- Discovery, Onboarding, Scaffolding, Endgame

This source is the main bridge from "Octalysis as theory" to "agent can diagnose a real behavior problem."

## 3. Section II: Marketing And Brand Cases

**Role**: case-pattern library.

Use it for:

- meaning/mission narrative patterns
- brand, ecommerce, loyalty, retention, referral, email, ARG, sales-training, and employee-training examples
- "loyalty is relationship, not reward maximization"
- social norm backfire and black-hat failure cases
- transferable vs non-transferable case conditions

Do not reduce it to a marketing skill. Most cases can be generalized to habits, learning, work, health, communities, AI agent workflows, and B2B adoption.

## 4. Section III: Strategy And Player Experience Design

**Role**: implementation flow engine.

Use it for:

- generalized strategy dashboard
- success metric ranking
- participant/motivation segment analysis
- target behavior and win-state definition
- feedback mechanism and trigger mapping
- reward/feedback mode selection
- behavior loop and self-reinforcing loop design

This is the source for turning diagnosis into an operational design workflow.

## Cross-Source Synthesis Rule

When sources differ in emphasis:

1. Use the book for terminology accuracy.
2. Use Section I for diagnostic wording.
3. Use Section II for case abstraction.
4. Use Section III for executable design process.

The final skill pack should speak in generic behavior-design language, not in product-growth or marketing-only language.
